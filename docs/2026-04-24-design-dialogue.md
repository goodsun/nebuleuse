# Nébuleuse 設計対話 議事録

**日付:** 2026-04-24
**形式:** Goodsun と AI アシスタントの対話
**目的:** 個人ナレッジプールの設計方針・技術選定・命名の決定

---

## 1. 出発点：Obsidian 的アプローチへの疑念

「自分の考えをまとめるためのツールとして Obsidian が流行っているが、これはどうか」という問いから出発。デビルズアドボケート的に以下を検討。

- **思考とノート取りの混同**：美しいノートやグラフを作る作業そのものが満足感を生み、本当の思考（矛盾と座る、前提を疑う）の代替品になる。
- **グラフビューの認知的錯覚**：ノード間のエッジは構文的関係であって意味的理解ではない。リンクを張ることで「繋がりを理解した気になって」思考が止まる。
- **ツール自体への最適化問題**：プラグインのカスタマイズが趣味化する。「良いノート環境ができたら考えが進む」という仮説はほぼ偽。
- **線形思考の回復という逆説**：散らばったアトミックノートを論として繋ぐ作業は別途必要で、Obsidian は助けてくれない。Luhmann 自身も Zettelkasten から本を書くときは手作業で順序を決めていた。
- **「後で見返す」神話**：脳内にインデックスがないノートは存在しないのと同じ。

結論：「考えをまとめる」目的での Obsidian 導入は手段の目的化になりやすい。本当の道具は「他人に読ませる前提で書くこと」と「議論」。

## 2. それでもナレッジプールは欲しい

批判されるべきは「整理行為そのものを思考と錯覚すること」であって、過去の自分の思考にアクセスできる状態を作ること自体は別物。

**有用なナレッジプールの条件：**
1. 書くコストが限りなく低い（分類は後回しか不要）
2. 検索で引ける（全文検索が速ければリンクもタグも不要）
3. 取り出しやすい場所で動く（摩擦ゼロ）

## 3. 「低レイヤーで実現したい」の解釈

「低レイヤー = 自分で全部把握できる、ブラックボックスがない」という意味であれば完全に妥当。レベル分け：

- **Lv1:** プレーンテキスト + 埋め込みインデックス（pgvector / sqlite-vss）
- **Lv2:** Lv1 + ローカル LLM で回答生成
- **Lv3:** 書き込みインターフェース統合（スマホ・音声・CLI）

「ツールを作ること自体の目的化」が最大の罠。最初は恥ずかしいほどシンプルに。

## 4. Bonsai + Chroma 構成の検討

最初、Bonsai を Elasticsearch マネージドサービス（bonsai.io）と取り違えた。Goodsun から **Bonsai-8B**（PrismML 発の 1-bit LLM、1.15GB、MLX 対応）の Qiita 記事を提示される。

AI アシスタントは当初「2026-03-31 リリース、訓練手法非公開、fork 版必須」などの状況証拠から「エイプリルフールネタ／生成記事」と疑ったが、Goodsun が実際にローカルで動かして対話ログを示し、実在を確認。

**反省点：** 知識カットオフ後（2026-01 以降）のモデルを「知らない」ことを「存在しない証拠」にしてしまった推論の誤り。

## 5. ベクター vs 全文 ≒ RAG vs エージェンティックの相似

Goodsun の指摘：「ベクター検索 vs 全文検索」と「RAG vs エージェンティック」は問いの構造が相似。両軸とも「人間が指示する度合い」と「機械に委ねる度合い」のトレードオフ。

- 全文検索 ↔ エージェンティック（人間／LLM がキーワードを決める）
- ベクター検索 ↔ RAG 1ショット（意味理解を機械に委ねる）

→ **SQLite 一本構成**が、2軸 × 2軸のどのセルにも自然に対応できる。LLM は「どんな SQL を書くか」だけで検索戦略を切り替えられる。

## 6. 技術選定の決定

| 項目 | 採用 | 却下した代案と理由 |
|---|---|---|
| ストレージ | SQLite（sqlite-vss + FTS5） | Chroma（別プロセス価値薄、日本語全文弱）／pgvector + pgroonga（Mac mini に重い） |
| 埋め込み | multilingual-e5 系（MLX） | — |
| LLM | Bonsai-8B（MLX、ローカル常駐） | RunPod の llm-jp-4 は重いタスクのエスカレーション先として保留 |
| 原本管理 | `~/nebuleuse/raw/YYYY/MM/DD-slug.md` プレーンテキスト | SQLite はインデックス（導出物）として扱い、壊れたら再生成 |
| 検索戦略 | ベクター + 全文の RRF ハイブリッド | — |

## 7. 命名

- bon-soleil（太陽） / CIEL（空） / Monolith（一枚岩） に並ぶ世界観
- 候補：Niwa / Hotaru / Mnemo / Étoile / Nébuleuse など
- 響きで **Nébuleuse**（星雲）に決定
- `nebulous`（英語：曖昧な）はネガティブな意味で却下
- フランス語の Nébuleuse は星雲という具体名詞で、CIEL（空）との連続性も完璧
- 含意：星雲の中に Bonsai が置かれている、拡散から結晶を取り出す動作の表現

## 8. リポジトリ Description

最終決定：
> Self-hosted knowledge base with SQLite-based vector + full-text search, driven by a local LLM.

`Bonsai-8B` 明示版も検討したが、モデル差し替えに備えて `a local LLM` と抽象化。具体実装は README で書く方針。

## 9. 最小構成の方針

```
~/nebuleuse/
  raw/                    # 原本（好きなエディタで書く）
  nebuleuse.db            # SQLite（vss + FTS5）
  bin/
    neb-ingest            # raw/ を監視して db 更新
    neb-search            # CLI 検索
    neb-ask               # 検索 → Bonsai に食わせて回答
```

**順序：** 道具より習慣が先。raw/ に書き溜める習慣作りと並行して、ingest → search → ask の3コマンドを最小実装。
